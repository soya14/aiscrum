# AI Scrum Team — Project Instructions

> **行為準則（Behavioral Guidelines）** 見本檔末節。這些準則適用於所有任務，偏向謹慎而非速度；對於簡單任務，自行判斷取捨。

## Design System

**所有 UI、前端、樣式相關的工作，必須以 `Design.md` 作為唯一設計規範來源。**

在進行任何以下操作之前，先讀取 `Design.md`：
- 新增或修改 HTML / CSS
- 選用顏色、字型、間距、圓角、陰影
- 設計新元件或修改現有元件
- 撰寫 inline style 或 CSS 變數

### 設計規範摘要（來自 Design.md）

| 項目 | 規範 |
|------|------|
| 設計系統 | Visitors — Analytical canvas vibrant spectrum |
| 主題 | light |
| 唯一字族 | OpenRunde（fallback: system-ui, sans-serif） |
| 主品牌色 | `#918df6` Radiant Violet |
| 主文字色 | `#181925` Slate Ink |
| 頁面背景 | `#ffffff` Canvas White |
| 邊框色 | `#e8e8e8` Light Gray |
| 最小圓角 | 8px（禁止直角） |
| 按鈕圓角 | 9999px（pill） |
| 卡片圓角 | 16px |
| 間距基底 | 4px |
| Section gap | 64px |

### 禁止事項

- 不引入 Design.md 未定義的新字族
- 不在大面積背景使用飽和色
- 不使用濃重不透明陰影（只用 `--shadow-subtle` 系列）
- 大字級不使用預設 `letter-spacing: normal`，必須套用 Design.md 指定的負值
- 不使用直角（0px border-radius）於任何互動元件或卡片

## 專案概覽

**AI Scrum Team 活動規劃 2026** — 單頁靜態網站，記錄 2026 年 4–10 月的六個月 AI Scrum 協作路徑圖。

### 檔案結構

```
/
├── index.html     # 主頁面（含所有 CSS 與 JS）
├── aimember.md    # O大 AI 成員提案說明（v3.1）
├── Design.md      # 設計系統規範（Visitors reference）
└── CLAUDE.md      # 本檔案
```

### 技術規格

- 純靜態 HTML/CSS/JS，無框架依賴
- 所有樣式以 CSS Custom Properties（`:root` 變數）管理
- 響應式：768px（平板）、600px（手機）兩個斷點
- 字型透過 Google Fonts 載入

### 開發原則

- 不引入 npm 套件或 build 工具（呼應末節「Simplicity First」）
- 直接編輯 `index.html`，保持單一檔案
- 新增樣式優先使用 Design.md 的 token，不自行定義新顏色

## 行為準則（Behavioral Guidelines）

減少常見 LLM 程式設計錯誤的通用原則。

### 1. 先思考再動手（Think Before Coding）

**不假設。不隱藏困惑。主動揭示取捨。**

實作前：
- 明確陳述你的假設。不確定時，先問。
- 若存在多種解讀，提出來讓使用者選——不要默默挑一個。
- 若有更簡單的做法，說出來，必要時推回需求。
- 若有任何不清楚之處，停下來，說明困惑點，然後提問。

### 2. 簡單優先（Simplicity First）

**用最少的程式碼解決問題。不寫推測性的功能。**

- 不做超出需求的功能。
- 不為只用一次的程式碼建立抽象層。
- 不做沒被要求的「彈性」或「可配置性」。
- 不處理不可能發生的錯誤情境。
- 若你寫了 200 行而 50 行可以搞定，重寫它。

自問：「資深工程師會說這太複雜嗎？」若是，簡化。

### 3. 精準修改（Surgical Changes）

**只碰必要的地方。只清理自己造成的亂。**

修改現有程式碼時：
- 不「順便優化」旁邊的程式碼、註解或格式。
- 不重構沒有壞掉的東西。
- 配合既有風格，即使你會用不同方式。
- 若發現無關的死碼，提出來——不要自行刪除。

當你的修改產生孤兒時：
- 移除「你的修改」所造成的未使用 import/變數/函式。
- 不移除原本就存在的死碼，除非被明確要求。

檢驗標準：每一行修改都應能直接追溯到使用者的需求。

### 4. 目標導向執行（Goal-Driven Execution）

**定義成功條件。循環驗證直到達成。**

將任務轉換成可驗證的目標：
- 「加入驗證」→「為無效輸入寫測試，再讓它通過」
- 「修 bug」→「寫一個重現 bug 的測試，再讓它通過」
- 「重構 X」→「確保重構前後測試都通過」

多步驟任務請先說明計畫：
```
1. [步驟] → 驗證：[檢查點]
2. [步驟] → 驗證：[檢查點]
3. [步驟] → 驗證：[檢查點]
```

強的成功條件讓你能自行迴圈收斂；弱的條件（「讓它能跑」）需要不斷澄清。

---

**這些準則有效的標誌：** diff 中更少不必要的修改、更少因過度複雜而重寫、澄清問題在實作前提出而非出錯後才問。

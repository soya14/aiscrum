安全運行 Codex

OpenAI 如何在企業環境中部署 AI 程式代理——沙盒隔離、審批策略、網路管控與代理原生遙測，構成一套平衡開發效率與安全管控的完整框架。

原文發布：2026 年 5 月 8 日 作者：OpenAI Security Team 翻譯整理：AI Scrum Team
核心目標

三個明確的部署目標

OpenAI 在內部部署 Codex 時設定了三大目標，確保 AI 程式代理在提升生產力的同時不引入安全風險。

目標 01
將代理限制在明確的技術邊界之內
目標 02
讓開發者能快速執行低風險操作
目標 03
將高風險操作明確化並要求審批
架構設計

雙層安全架構

Codex 的安全控制來自兩個協同運作的層次，共同界定代理「能做什麼」與「何時需要暫停」。兩層共同運作，缺一不可。

第一層
沙盒模式
定義代理的技術能力邊界——可以寫入哪些位置、是否能存取網路、執行哪些模型生成的命令。作業系統層級的強制機制保障這些限制無法被繞過，預設狀態為無網路存取、寫入權限僅限於活躍工作目錄。
第二層
審批策略
決定代理在執行操作前何時必須暫停並詢問使用者——例如離開沙盒、使用網路，或執行不在信任命令列表中的指令。讓人類保持對高風險決策點的最終控制。
安全控制

四大控制面向

OpenAI 從認證存取、沙盒隔離、網路管控、Shell 指令安全四個維度，構建完整的 Codex 安全部署框架。

🔐
認證與存取控制
CLI 與 MCP 的 OAuth 憑證儲存在作業系統安全金鑰環中，登入強制透過 ChatGPT 進行，存取範圍限制在 ChatGPT Enterprise 工作區內，確保 Codex 使用行為綁定工作區層級的管控。
  • OAuth 憑證存入 OS 安全金鑰環，不暴露於環境變數
  • 強制透過 ChatGPT 登入驗證，統一身份管理
  • 存取範圍綁定 ChatGPT Enterprise 工作區
📦
沙盒隔離
本地端使用作業系統強制沙盒，將代理操作範圍限制在當前工作目錄內；雲端則在獨立的 OpenAI 管理容器中執行，完全無法存取宿主系統或其他資料。
  • 本地：OS 層級強制沙盒，寫入範圍限於工作目錄
  • 雲端：隔離容器部署，與宿主系統完全隔絕
  • 預設：網路存取關閉，寫入權限限於活躍工作區
🌐
網路政策管控
代理預設關閉所有對外網路存取。OpenAI 採用受管理的網路策略：允許已知目標端點、封鎖不想讓 Codex 存取的目標、對陌生域名要求審批——讓代理在不授予廣泛網路權限的情況下完成常見工作流程。
  • 預設狀態：所有對外網路連線關閉
  • 允許清單：已驗證且必要的目標端點
  • 封鎖清單：明確禁止存取的目標
  • 陌生域名：觸發審批流程，不自動放行
⚡
Shell 指令安全分類
Codex 不把每條 Shell 指令視為同等安全。日常開發常用的無害指令可在沙盒外直接執行，而特定危險指令則被封鎖或要求審批——讓代理能快速完成常規任務，同時對高風險操作施加必要管控。
  • 常用無害指令：依規則分類，無需逐一審批
  • 危險指令：封鎖或強制審批，防止意外執行
  • 規則驅動分類，而非手動維護白名單
可觀測性

代理原生遙測

OpenAI 保留完整的代理原生遙測數據，以便稽核代理的每一步行為。Codex 支援透過 OpenTelemetry (OTel) 選擇性開啟監控，協助團隊進行稽核、問題調查與法規遵循，且不會削弱本地安全預設值。

「我們的 AI 安全分類代理使用 Codex 日誌來檢視原始請求、工具活動、審批決策、工具結果，以及相關的網路策略決定或封鎖事件。」
— OpenAI Security Team，Running Codex Safely（2026/05/08）

OTel 日誌記錄的事件類型：

使用者提示 工具審批決策 工具執行結果 MCP 伺服器使用 網路代理允許事件 網路代理封鎖事件
企業合規支援
Enterprise 與教育版客戶可透過 OpenAI Compliance Platform 存取 Codex 活動日誌，滿足企業級合規要求。遙測功能預設關閉，需在設定中明確啟用，確保資料收集行為符合預期。
結論

讓安全團隊更有把握地啟用 Codex

Codex 提供了控制介面、配置管理、沙盒隔離與詳細的代理感知遙測，為企業安全採用 AI 程式代理奠定基礎。安全團隊可在維持必要的可視性與企業安全管控的同時，讓開發者充分享有生產力提升。

🚀
開發者受益
快速執行低風險操作，高風險操作有明確審批流程，日常開發節奏不受干擾。
🛡️
安全團隊受益
完整稽核軌跡、可設定的網路策略與審批規則，滿足企業安全與合規要求。

安全運行 Codex

OpenAI 在真實工作流程中治理程式代理所使用的控制、邊界與遙測機制全面解析。

作者：OpenAI Security Team 翻譯整理：AI Scrum Team · 2026/05/13

隨著 AI 系統能力不斷提升，它們越來越多地代替使用者執行任務。程式代理可以自主審查程式庫、執行命令、與開發工具互動——這些都是過去需要人工直接操作的工作。

在設計 Codex 的這些能力時，我們同步設計了組織安全部署所需的控制手段。安全團隊需要治理代理運作的方式：代理可以存取什麼、何時需要人工審批、可以與哪些系統互動，以及存在哪些遙測可以解釋代理的行為。

在 OpenAI，我們帶著幾個明確目標部署 Codex：將代理限制在明確的技術邊界之內、讓開發者在低風險操作上快速行動，以及將高風險操作明確化。我們同時保留代理原生遙測，以便理解與稽核代理的行為。在實務上，這意味著受管理的設定、受限的執行環境、網路政策，以及代理原生日誌。

目標 01
將代理限制在明確的技術邊界之內
目標 02
讓低風險的日常操作保持順暢無阻
目標 03
保留代理原生遙測以便理解與稽核
控制機制

控制 Codex 的運作方式

我們以一個簡單的原則部署 Codex：在受限環境內保持高效生產力、低風險的日常操作應無摩擦地執行，而高風險操作則應暫停等待審查。

沙盒與審批

審批與沙盒協同運作。沙盒定義了技術執行邊界，包括 Codex 可以寫入的位置、是否可以存取網路，以及哪些路徑受到保護。審批策略決定 Codex 何時必須請求執行某個動作，例如當它需要執行沙盒以外的操作時。使用者可以一次性批准，或批准該類型操作在本次 session 內自動允許。

對於常規審批請求，我們使用 Auto-review 模式——此功能開啟後，可自動批准特定類型的請求，減少使用者需要手動停下來審批的頻率。Codex 將計劃執行的動作與近期上下文傳送給自動審批子代理，後者可自動批准低風險動作，不打斷使用者工作流程；對高風險或可能有意外後果的動作，則仍會暫停等待審查。

第一層
沙盒模式
定義代理的技術能力邊界——可寫入哪些位置、是否能存取網路、執行哪些模型生成的命令。
第二層
審批策略
決定代理何時必須暫停並詢問使用者，例如離開沙盒、使用網路或執行信任清單外的指令。
config.toml & requirements.toml TOML 
# config.toml

# 開啟自動審批模式
approvals_reviewer = "auto_review"
# 自動將已知開發目錄加入沙盒寫入範圍
sandbox_workspace_write.writable_roots = ["~/development"]

# requirements.toml

# 要求 Codex 在沙盒模式內運作
allowed_sandbox_modes = ["read-only", "workspace-write"]

網路存取

我們不讓 Codex 擁有不受限制的對外存取權限。我們的受管理網路政策允許已知目標端點、封鎖我們不想讓 Codex 存取的目標，並對陌生域名要求審批。這讓 Codex 能夠完成常見的既有工作流程，而不需要授予廣泛的網路存取權限。

requirements.toml TOML 
# requirements.toml

# 確保網頁擷取只來自 OpenAI 的快取
allowed_web_search_modes = ["cached"]

[experimental_network]
# 開啟網路代理
enabled = true
# 允許 Codex 與本機連接
allow_local_binding = true
# 封鎖對此域名的所有請求
denied_domains = ["pastebin.com"]
# 自動允許對這些域名的請求
allowed_domains = ["login.microsoftonline.com", "*.openai.com"]

身份與憑證

我們也管理 Codex 的身份驗證方式。CLI 與 MCP 的 OAuth 憑證儲存在作業系統安全金鑰環中，登入強制透過 ChatGPT 進行，存取範圍固定在我們的 ChatGPT Enterprise 工作區。這讓 Codex 使用行為綁定工作區層級的管控，並使 Codex 活動日誌可透過 ChatGPT Compliance Logs Platform 供企業工作區存取。

config.toml TOML 
# config.toml

# 將 CLI 驗證憑證儲存在 OS 金鑰環
cli_auth_credentials_store = "keyring"
# 將 MCP 憑證儲存在 OS 金鑰環
mcp_oauth_credentials_store = "keyring"
# 強制透過 ChatGPT 驗證
forced_login_method = "chatgpt"
# 強制驗證至特定 ChatGPT 工作區
forced_chatgpt_workspace_id = "<workspace-uuid>"

指令規則

我們使用規則，讓 Codex 不會把每條 Shell 指令都視為同等安全。工程師在日常開發中常用的無害指令，可在沙盒外直接執行；特定危險指令則被封鎖或要求審批。這讓 Codex 能快速完成一般工程任務，同時對我們不希望在沙盒外執行的指令模式強制審查或封鎖。

default.rules Rules DSL 
# default.rules

prefix_rule(
    pattern       = ["gh", "pr", ["view", "list"]],
    decision      = "allow",
    justification = "允許透過 gh CLI 進行唯讀的 GitHub PR 檢視。",
)
prefix_rule(
    pattern       = ["kubectl", ["get", "describe", "logs"]],
    decision      = "allow",
    justification = "允許 Kubernetes 資源檢視以供除錯使用。",
)

受管理的設定

我們透過雲端受管理的 requirements、macOS 受管理偏好設定，以及本地 requirements 檔案的組合來套用上述安全策略。Requirements 是管理員強制執行的控制，使用者無法覆蓋。macOS 受管理偏好設定與本地 requirements 檔案讓我們能維持一致的基準，同時仍可依團隊、使用者群組或環境測試不同設定。這些設定適用於所有本地 Codex 介面，包括桌面應用程式、CLI 與 IDE 擴充套件。

可觀測性

代理原生遙測與稽核軌跡

控制只是工作的一半。代理部署後，安全團隊需要掌握代理正在做什麼以及為何這樣做的可視性。傳統的安全日誌在查看 Codex 所採取的動作時仍然有用，但它們主要回答的是「發生了什麼」：某個程序啟動了、某個檔案被修改了、某個網路連線被嘗試了。防禦者仍然需要自行判斷 Codex 為何執行某件事，或使用者的意圖是什麼。

Codex 可以為安全團隊提供更具代理意識的視角。Codex 支援透過 OpenTelemetry 匯出各種 Codex 事件的日誌。Enterprise 與 Edu 客戶也可透過 OpenAI Compliance Platform 存取 Codex 活動日誌。

OTel 日誌記錄的事件類型：

使用者提示 工具審批決策 工具執行結果 MCP 伺服器使用 網路代理允許事件 網路代理封鎖事件
config.toml — OTel 遙測設定 TOML 
# config.toml

[otel]
log_user_prompt = true
environment     = "prod"

[otel.exporter.otlp-http]
endpoint = "http://localhost:14318/v1/logs"
protocol = "binary"
「我們的 AI 安全分類代理使用 Codex 日誌來檢視原始請求、工具活動、審批決策、工具結果，以及相關的網路策略決定或封鎖事件——再將分析結果呈報安全團隊，以區分預期的代理行為、無害的錯誤，以及真正需要升級處理的活動。」
— OpenAI Security Team，Running Codex Safely（2026/05/08）

我們同時也將相同的遙測用於日常運營：了解內部採用趨勢的變化、哪些工具與 MCP 伺服器被使用、網路沙盒封鎖或提示的頻率，以及部署在哪些方面仍需調整。這些 OpenTelemetry 日誌可集中整合至 SIEM 與合規日誌系統。

展望

讓安全團隊更有把握地啟用 Codex

隨著 Codex 等程式代理日益融入開發工作流程，安全團隊需要專為管理這一轉變而設計的工具。Codex 提供了確保安全採用所需的控制介面、設定管理、沙盒隔離，以及詳細的代理感知遙測。有了這些能力，安全團隊可以更有把握地啟用 Codex，在開發者生產力與企業安全所需的可視性及管控之間取得平衡。
